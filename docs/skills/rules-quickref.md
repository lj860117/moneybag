# 关键规定速查

> 从 hermanlei-conventions 提炼的核心要点，避免每次都要读 1200 行。
> 完整版见 SKILL.md 第三章。

## 🔴 铁律（绝对不能违反）

1. **绝不用正则做 C# 批量重构** — 手工逐文件替换
2. **改代码前先确认可回滚** — 告知用户即将修改的文件清单
3. **改完一个文件立即验证** — 不攒改动
4. **超过 2 轮修不好就停** — 告诉用户需要回滚，不越修越烂
5. **linter ≠ 编译器** — 不以 VSCode 零错误作为"修好了"的依据
6. **涉及技术深度先查参考** — 不凭空写算法/复杂工具
7. **方案要有出处** — 关键决策说得出"谁用过"
8. **最小可用版本先交** — 核心功能先跑通再迭代
9. **超出能力范围就说** — 建议用现成库
10. **记忆 ≠ 事实** — MEMORY.md 和日志中的信息可能过时，涉及路径、配置、API 变更时必须先验证再行动
11. **删除系统/个人文件必须三确认** — ① 列出完整文件清单 ② 说明风险 ③ 用户确认后才执行。走回收站不走 rm，每批 ≤10 个文件。系统目录（`/`、`/System`、`Library`、`~/.config`、`C:\Windows`、`C:\Program Files`）绝不碰
12. **改完必须只读验证（verification-gate）** — 改完代码后、报告用户前，必须只读重检：① 重读所有改动文件确认写入正确 ② 检查关联文件是否遗漏 ③ 检查逻辑/拼写错误。不改文件，只验证。未通过 → 修完再过 gate
13. **"不能"=穷尽所有选项** — 某事不工作时，先尝试至少 3-5 种替代方案（搜索、换工具、查记忆中的类似案例）再考虑说"做不到"。第一次失败不叫"不能"，叫"还没找到方法"
14. **SKILL.md 只放规则，知识靠引用** — 操作手册/模板/示例/知识索引放 `references/`，SKILL.md 留引用指针。新增前先判断：规则 vs 知识
15. **SKILL.md 写入前必须预算检查** — `wc -l`：≤1300 OK / 1301~1500 警告 / >1500 禁止写入，先瘦身
16. **Git 推送分两档** — ① 用户说"收工/去睡了/今天结束"→ **主动提醒**做完整同步（Skill + Memory + 日志 + 保密审计） ② 用户说"更新 git/push"→ 只推 Skill 改动，不动 Memory
17. **启动清单是硬门禁，不是建议** — 不管会话怎么开始（"继续"/"都搞了"/"新任务"），**收到第一条实质消息时必须先跑启动清单**。在启动清单完成之前，禁止执行任何代码编写、文件修改、git 操作。违反此条 = P0 事故。来源：2026-04-12 V5.0 事故（跳过启动清单 → 4个功能模块未经规范检查直接 push 生产）
18. **后端 API 做了必须验证前端调了** — 每次新增后端 API 后，必须 grep 前端代码确认有调用。后端做了前端没接 = 白做。来源：2026-04-13（政策5主题/AI点评/持仓智能做了后端，前端 0 次调用）
19. **多 ID 系统必须统一入口函数** — 项目里如果有多个 ID（userId/profileId/wxworkId），必须统一为一个入口函数，所有 API 调用都走同一个。两个函数返回不同 ID = 数据存错地方。来源：2026-04-13（getUserId vs getProfileId 导致 18 处 API 数据隔离破裂）
20. **企微/第三方推送先查消息格式兼容性** — 接入任何推送渠道前，先查目标端支持什么格式（纯文本/Markdown/HTML）。不要假设支持 Markdown。来源：2026-04-13（企微 Markdown 在微信端显示"暂不支持此消息类型"）
21. **压缩/单行 JS 禁止用 replace 插大段代码** — 对压缩过的 app.js，不能用 `replace_in_file` 插入多行 HTML 模板或新函数，极易截断现有函数体导致白屏。正确做法：以能跑的旧版为基底，用脚本（Node/Python）精确打补丁，每个补丁后 `node --check` 验证。来源：2026-04-17（replace 截断 doAddStock → await 泄漏全局 → 白屏 P0 事故）
22. **FastAPI 路由：具名路径必须定义在通配路径之前** — `/api/user/preference` 必须在 `/api/user/{user_id}` 之前定义，否则 FastAPI 按顺序匹配会把 `preference` 当成 `user_id`。来源：2026-04-17（偏好 API 返回了空用户数据而非偏好配置）

## 🔴 P0 事故档案

> 重大事故教训存档。每条铁律的来源事故记录在此，确保后人知道"为什么有这条规则"。

### #001：V5.0 启动清单跳过事故（2026-04-12）

- **影响范围**：V5.0 全部 4 个新功能（风控/配置/选基/选股）+ 14 模块拆分，共 2000+ 行代码未经规范检查 push 到生产
- **根因**：会话恢复时（"请继续完成未完成的任务"），AI 直接继续写代码，完全跳过启动清单 5 个步骤
- **跳过的关键步骤**：① 未读取 SESSION-STATE / MEMORY / 日志 ② 未加载 hermanlei-conventions Skill ③ 未执行 verification-gate ④ 量化逻辑未先调研 ⑤ 方案无出处
- **产生的铁律**：#17（启动清单硬门禁）
- **防御机制**：铁律 #17 + SOUL.md 硬指令双保险

### #002：Phase 0 前端白屏事故（2026-04-17）

- **影响范围**：部署后用户手机白屏，回滚后恢复
- **根因**：用 `replace_in_file` 匹配 `doAddStock` 函数第一行时，只匹配了 `{...return}` 部分，后续代码（`const cost=...` 等）被新插入的 `runStockAnalysis` 函数截断，导致 `await` 泄漏到全局作用域，浏览器 JS 引擎报错整个脚本停止
- **为什么 node --check 没抓到**：花括号数量碰巧因为新增函数补回了平衡（2541/2541），语法层面合法，但语义上 `doAddStock` 不完整
- **修复方式**：以服务器旧版为基底，用 Node 脚本精确打 5 个补丁，每步验证
- **产生的铁律**：#21（压缩 JS 禁止 replace 插大段代码）
- **附带发现**：#22（FastAPI 路由顺序问题）

## 🔴 Skill 质量门禁（创建/修改自建 Skill 时必须逐项检查）

> 来源：2026-04-12 全量审计教训（81→95 分）。以后新建或修改 Skill 都过一遍这个清单。

| # | 检查项 | 不合格表现（审计中发现的） | 合格标准 |
|---|--------|--------------------------|---------|
| 1 | **description 格式** | 写成普通介绍句 | 必须以 `Load when:` 开头 + 编号列举触发条件 + `NOT needed for:` 排除项 |
| 2 | **Output Format 区块** | 缺失 | 自建 Skill 必须有 `## Output Format` 区块，定义输出结构 |
| 3 | **version 字段** | 缺失 | frontmatter 必须有 `version: x.y.z`（语义化版本） |
| 4 | **references/ 无重复** | 同一文件在多个 Skill 的 references/ 中重复存在 | 文件只存在于归属 Skill 中，其他 Skill 用路径引用 |
| 5 | **session-log.md** | 缺失 | 自建 Skill 必须有 `session-log.md` 模板 |
| 6 | **行数预算** | >500 行 | SKILL.md ≤500 行（见 MEMORY.md 瘦身铁律） |
| 7 | **第三方 Skill 不动** | 修改了市场装的 Skill | 第三方 Skill（ClawHub/Skill Stack/市场）只调用不魔改 |

**执行时机**：每次 `git add` Skill 文件前，对照此表逐项过。

## 🟡 Memory vs Skill 分类标准（事实 vs 方法）

> 借鉴 Hermes Agent 的设计：**Memory 存"事实"，Skill 存"方法"**。边界不清会导致信息散乱。

| 存什么 | 存哪里 | 判断标准 | 示例 |
|--------|--------|---------|------|
| **事实** | MEMORY.md / 日志 / preferences.md | 背景信息、一次性决策、个人偏好、项目状态 | 项目路径、Git 身份、用户习惯、技术选型决策 |
| **方法** | SKILL.md / references/ / 独立 Skill | 可复用流程、标准操作步骤、规则模板 | 闭环流程、代码体检清单、changelog 生成规范 |
| **踩坑** | 先写日志 → 提炼为铁律/规则（Skill） | 教训如果可泛化为"以后都要这样做" → Skill | GBK 编码坑 → 铁律 |
| **灵感** | 日志 `## 灵感`（仅 Memory） | 还未验证的想法，不写入 Skill | "可以做 XX 功能" |

**三问速判**：
1. 这条信息是"知道就行"还是"要按这个做"？ → 前者 Memory，后者 Skill
2. 换个会话/换个 AI，这条还需要遵守吗？ → 需要 → Skill
3. 只跟当前状态有关、以后可能变？ → Memory

## 🟡 关键规范

- **Git commit 格式**：`[home] 类型: 简短描述`
- **修改 Skill 必须告知用户**
- **中断任务必须记录** — 写入 MEMORY.md 待恢复任务区
- **注释用中文**
- **禁止模糊文件命名** — `记录1`、`想法`、`杂项`、`今天的东西`、`temp`、`未命名` 一律禁止，详见 `daily-log-template.md`
- **运行中识别关键变化** — 踩坑/灵感/决策要立即记录，不要等做完才回忆，详见 `startup-checklist.md` 触发记录表
