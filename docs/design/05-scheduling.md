> **何时读**：做凌晨工厂相关、cache 策略、LLM 调度、白天 API 响应时序时。

## 🔗 模块契约（M1 W1 前：文档约束 / M1 W1 后：见 Protocol）

**上游（本模块消费谁的输出）**：
- `03-rule-engine.md` → 凌晨工厂批量调用 `rule_engine.compute_all()`
- `04-ai-interface.md` → 凌晨工厂批量调用 `gateway.call()` 生成文案
- `infra/data_source` → Tushare / AKShare 数据采集
- `infra/cache` → 三层缓存（A/B/C）底层实现

**下游（谁消费本模块的输出）**：
- 所有 API 读接口 → 先读 precomputed_cache，未命中才实时算
- 前端时效标注 → 每个 UI 区域必须标 A/B/C 层来源
- 企业微信推送 → 早安简报从本模块产出

**改动本模块前必须评估**（见 ANCHOR 改动传播表）：
- **三层缓存 key / TTL** → 所有使用该 key 的 use_case / api（否则读不到数据）
- **event 触发器**（A 层即时重算）→ `06` 交易录入 + `07` 清单触发
- **凌晨工厂 pipeline 阶段划分** → checkpoint 文件格式 + 人工补跑命令
- **失败告警规则** → 企业微信推送配置 + `api_health_check`
- **紧急模式触发条件** → 前端 UI 按钮 + cache_warmer 盘中逻辑

**关键不变式**：
- LLM **不在用户关键路径**（白天读缓存）
- **事实和翻译分离**（A 层纯 Python / C 层 LLM）
- **凌晨算一次，白天读一天**（除强烈事件外不调 LLM）
- **按模型分桶**（V3 日限 200，R1 日限 20）

**M1 W1 后**：缓存接口走 `CacheProtocol`（三层通过不同实现），工厂流程 use_case 化。

## 一、核心理念

**"AI 只翻译事实"——事实是昨晚就能算好的。**

旧架构让 LLM 白天现场算命 → 新架构让它凌晨批量翻译 + 白天柜台展示。
**既发挥模型擅长，又省钱秒开。**

## 二、凌晨工厂 + 白天柜台

```
┌──── 凌晨 01:00-08:30（后台 night_worker，慢工出细活）────┐
│                                                        │
│  01:30  数据采集（Tushare / AKShare）                    │
│    ↓                                                   │
│  02:00  规则引擎批量计算（无 AI，纯 Python）              │
│         - 每用户配置偏离度                               │
│         - 集中度 / 行业敞口 / 浮亏触发                   │
│         - 5 维评分 TOP N、候选池                         │
│         → 结果写 precomputed_cache                      │
│    ↓                                                   │
│  02:30  LLM 批量翻译（大头）                             │
│         a. V3 × N: 每用户配置体检白话版                  │
│         b. V3 × 1: 市场环境总结                          │
│         c. R1 × 1-2: 昨日决策归因 / 冲突仲裁             │
│         → 结果写 precomputed_cache + llm_gateway 磁盘缓存│
│    ↓                                                   │
│  07:30  生成早安简报 → 推送企业微信                       │
│                                                        │
└────────────────────────────────────────────────────────┘
                          ↓
┌──── 白天（API 柜台，秒级响应）─────────────────────────────┐
│                                                        │
│  用户打开 App                                            │
│    ↓                                                   │
│  API 先读 precomputed_cache（TTL 4-8h） → 命中直接返回    │
│    ↓                                                   │
│  未命中才实时算（罕见：盘中强制再平衡事件）                 │
│    ↓                                                   │
│  llm_gateway 1h 缓存兜底，避免重复调用                    │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 三、落地改动清单

| 项 | 现状 | 新架构改动 |
|---|---|---|
| `night_worker.py` 调用链 | `→ decision_maker.generate_decisions`（R1 出 buy/sell）| `→ rule_engine.compute_all(user) → decision_maker_v2.generate_interpretation`（V3 翻译事实）|
| `precomputed_cache._PRECOMPUTED_TTL` | 只有 recommendations / decisions / signal 等 | **新增** allocation_checkup(8h) / portfolio_health(4h) / risk_inventory(4h) / attribution_report(24h) / candidate_pool(4h) |
| `llm_gateway` 限额 | 单桶 DAILY_LIMIT=100 | **按模型分桶**：V3 日限 200、R1 日限 20 |
| `cache_warmer.py` 盘中刷新 | 刷数据层 | **只刷事实层（规则引擎），不调 LLM**；仅偏离 >15% 时触发一次 V3 生成告警文案 |
| 冷启动 | 新用户无凌晨数据 → 报错 | 首次请求触发一次同步计算（规则引擎 <1s + V3 <3s），之后自动入凌晨队列 |

## 四、成本与体验对比

| 维度 | 旧架构 | 新架构 |
|---|---|---|
| LLM 日成本 | ¥0.45-1.5（白天现调）| ¥0.30-0.50（凌晨批量，缓存命中率 >80%）|
| 白天首屏 | 30-50 秒（等 R1）| 1-3 秒（读磁盘）|
| LLM 挂时 | 用户看不到分析 | 读昨晚缓存 + 规则引擎兜底，不感知 |
| 降级路径 | 依赖 `_rule_based_decision` | 天然存在：缓存 → 规则引擎 → 空壳报告 |

---

## 五、关键原则

1. **LLM 不在用户关键路径**：白天读缓存，LLM 挂了不影响看报告。
2. **事实和翻译分离**：规则引擎白天必须能独立跑出结果，LLM 的翻译只是锦上添花。
3. **凌晨算一次，白天读一天**：除非强烈事件（偏离 >15% / 单票浮亏 >25%），白天不触发 LLM。
4. **按模型分桶计费**：V3 便宜管够，R1 金贵谨慎。

## 六、失败处理与兜底（凌晨工厂挂了怎么办）

凌晨工厂依赖 Tushare / AKShare / DeepSeek 三个外部服务，任何一家晚上抽风整条流水线就断。必须三层兜底：

### 6.1 第一层：前置健康巡检（1:00-1:30）

`night_worker.py` **第一步就跑 `datasource_health_check.py`**（已有脚本）：

- 任一数据源连接失败 → 立刻企业微信告警 + **不继续往下跑**（避免产出半成品污染缓存）
- DeepSeek 不可用 → 仍跑数据采集和规则引擎，跳过 LLM 翻译阶段（白天用规则引擎兜底文案）
- 每次巡检结果写 `data/night_worker/health_{date}.json`

#### 数据源三级降级链（含 AKShare 失效监控）

```
主源      → Tushare Pro（规范、稳定、基本面全）
第一降级  → AKShare（另类数据覆盖广；⚠️ 爬虫聚合，接口易失效）
第二降级  → baostock（纯 A 股日线、免费免注册、稳定）
```

**监控规则**：
- **AKShare 接口失效率**：每周统计失败率，>20% 触发企业微信告警（爬虫聚合类数据可能某天就挂）
- **关键指标监控**：北向资金 / 融资融券 / 龙虎榜等另类数据接口变化最频繁
- **降级决策**：只做日线判断时，Tushare 挂了**可直接跳 AKShare 去 baostock**，不硬等 AKShare

> 📌 **不上 pytdx / 通达信逆向协议** —— 合规风险，你项目用不上高频。
> 📌 **不考虑 L2 / WebSocket / tick 级** —— 你是日线规则判断，不做盘中实时。

### 6.2 第二层：分阶段 checkpoint（可断点续跑）

把 night_worker 切成 **3 个独立阶段**，每阶段完成写 checkpoint：

```
阶段 1: 数据采集    → data/night_worker/checkpoints/{date}_stage1_data.json
阶段 2: 规则引擎     → data/night_worker/checkpoints/{date}_stage2_rules.json
阶段 3: LLM 翻译    → data/night_worker/checkpoints/{date}_stage3_llm.json
```

失败后人工补跑，跳过已完成阶段：

```bash
python night_worker.py --resume --from-stage=llm   # 只重跑 LLM 翻译
python night_worker.py --resume --from-stage=rules # 重跑规则+LLM
```

### 6.3 第三层:白天兜底（用户不感知失败）

- 凌晨没跑 / 跑失败 → API 读取时 `precomputed_cache` 用**前一天**的数据
- 前端顶部加一个**黄色横幅告警**："数据截至 XX 月 XX 日 XX:XX，今日数据更新失败"
- 规则引擎**即时重算**（数据新但 LLM 文案旧，优先保证规则判断是新的）
- 绝不让用户看到 500 错误或空白页

### 6.4 监控与告警

| 触发条件 | 动作 |
|---------|------|
| 01:30 健康巡检失败 | 企业微信 @ 你 |
| 03:30 仍未完成阶段 2 | 企业微信 @ 你（超时）|
| 08:30 早安简报未推送 | 企业微信 @ 你 + `api_health_check` 加一项"昨夜状态" |
| 连续 2 天失败 | 升级告警（短信/邮件）|

**原则**：**人工介入的门槛越低越好，但必须告诉你出了什么事、卡在哪一步**。不要让工厂静悄悄挂掉，白天才发现。

## 七、三层缓存策略（实时性关键）

§二 的"凌晨工厂 + 白天柜台"解决"首屏秒开"，但不覆盖**盘中实时场景**——比如用户刚卖掉一只股票，回到主页如果还显示凌晨的集中度数据，就是错的。

**核心改造**：把缓存显式切成 **3 层**，分层更新，分层展示。

### 7.1 三层定义

| 层 | 名称 | 更新频率 | 内容 | 触发机制 |
|----|------|---------|------|---------|
| **A** | 实时事实层 | 秒级 | 偏离度 / 集中度 / 浮亏 / 行业敞口 / 应急金月数 | 用户交易 / 持仓变动 **event hook 即时算** |
| **B** | 准实时数据层 | 分钟级（交易时段）| 行情、估值分位、恐贪指数、北向资金 | `cache_warmer` 盘中定时 + 延迟数据源 |
| **C** | 隔夜解读层 | 小时/天级 | LLM 文案、复盘归因、候选池、体检报告文字版 | 凌晨工厂批量 |

### 7.2 为什么分三层

- **A 层（事实）**：用户做决策看的是"我现在偏离多少"，**必须秒级准确**，但它是**纯 Python 计算**（输入有了结果立刻出），零成本。
- **B 层（行情）**：行情几分钟变一次，不需要实时推送到每个用户，用定时刷新够了。
- **C 层（文案）**：LLM 写的"你应急金不够"这句话**原理是隔夜仍然成立的**，只要 A 层事实还满足触发条件，文案就复用昨晚的。成本大头在这层，必须缓存。

**关键洞见**：**事实会变，但原理不变**。规则引擎昨晚的触发项今天仍然存在 → LLM 文案就不用重算。

### 7.3 Event-driven 即时重算（A 层）

触发即时重算的事件：

| 事件 | 重算内容 | 触发源 |
|------|---------|-------|
| 用户录入/编辑交易 | 持仓金额、集中度、行业敞口、浮亏 | `/api/portfolio/transaction` 成功后 |
| 用户导入券商流水 | 批量同上 | `/api/portfolio/import` 成功后 |
| 资产负债表字段变更 | 总资产、偏离度、应急金 | `/api/family/balance-sheet/update` 后 |
| 盘中大幅波动（单票 ±8%）| 该票的浮亏、触及止损规则 | cache_warmer 检测到后 |

**实现**：

```python
# infra/events.py
@on_event("portfolio.transaction.created")
def recalc_realtime_facts(user_id, transaction):
    # 纯 Python，<1 秒
    facts = rule_engine.compute_facts(user_id)
    cache.set(f"facts:{user_id}", facts, ttl=300)
    # 如果触发了新的规则（比如集中度刚超过 25%），立即推一条 websocket
    for alert in rule_engine.check_triggers(facts):
        push_alert(user_id, alert)
```

### 7.4 前端展示的三层对应

| UI 区域 | 读哪层 | 时效标注 |
|--------|-------|---------|
| 顶部"我的家底"卡片（总资产/偏离度）| A | 实时 |
| "持仓"页的浮亏 / 集中度 | A | 实时 |
| "市场"页的估值 / 恐贪 / 北向 | B | "更新于 X 分钟前" |
| "体检报告"页的 AI 文案 | C | "生成于今晨 X:XX" |
| "候选池" / "方向" | C | "更新于今晨" |

**规则**：每个 UI 区域**必须明确标注**它的时效层级，不让用户误以为所有数字都是实时的。

### 7.5 盘中黑天鹅场景（极端实时）

用户盘中想紧急调仓 → 数据必须是最新的。

**应对**：
- "**紧急模式**"按钮：用户在 UI 上点一下，触发全量 A 层重算 + 强制刷 B 层（同步调用数据源不走缓存）
- 极端市场事件（沪深300 日内跌 >4%）→ cache_warmer 自动进入"紧急模式"刷新节奏
- 这种场景下**也不调 LLM**——紧急时刻只看事实数字，不看文案

### 7.6 成本对比

| 场景 | 旧（单层缓存）| 新（三层缓存）|
|------|-------------|-------------|
| 用户录入交易 → 看偏离度 | 读凌晨缓存（错的）| A 层即时算（对）|
| 盘中看行情 | 读凌晨缓存（错的）| B 层分钟级（对）|
| 看 AI 体检文案 | 读凌晨缓存（对）| C 层凌晨缓存（对）|
| 凌晨工厂挂了 | 白天数据全错 | **只 C 层降级**，A 层 B 层正常 |
| LLM 日成本 | 盘中怕不准要重调 | **盘中不调 LLM**，成本不增加 |

**结论**：三层策略**不增加 LLM 成本**（C 层沿用凌晨批量），但解决实时性问题。
