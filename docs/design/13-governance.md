## 🔗 模块契约

**本文件是治理规则的中枢**，CI 规则、PR template、季度体检指标都从这里派生。

**上游（本文件依据谁）**：
- `00-ANCHOR.md` → 11 条不变式是 CI 规则的源头
- `12-framework-refactor.md` → 四层架构是 import-linter 规则的源头
- 其他模块文件的"关键不变式" → grep 级 CI 规则

**下游（谁依赖本文件）**：
- CI / pre-commit 配置 → 实现本文件定义的规则
- PR template → 引用本文件 §十二
- `docs/ARCHITECTURE_HEALTH.md`（每季度更新）→ 引用本文件 §八 指标

**改动本模块前必须评估**：
- **CI 规则变动** → 同步 `00-ANCHOR.md` 不变式（不能出现"文档有 CI 没有"）
- **PR template 改动** → 所有开发者 / 协作 AI 的工作流
- **季度体检指标新增** → 同步 `10-roadmap.md` milestone 验收 + `11-risk-metrics.md`

**关键不变式**：
- **CI 拦的 = 不变式落地** —— 如果不变式有 CI 跟上，形同虚设
- **PR 三件套**：单测 + ARCHITECTURE.md + CHANGELOG.md
- **两次就抽象**（第一次写允许，第二次必抽）

## 一、核心理念

**一次重构是"战役"，保持结构合理是"治理"。**

靠自律不如靠工具，靠文档不如靠 CI。

## 二、被动防御：CI / Linter 清单（一次配好、自动拦截）

最省力的防线。一次配好，以后每次 PR 自动跑。

### 2.1 文件行数上限

```yaml
# scripts/check_file_sizes.py 或 pre-commit hook
rules:
  backend/main.py:                 max_lines: 200
  backend/api/*.py:                max_lines: 400
  backend/use_cases/*.py:          max_lines: 300
  backend/domain/services/*.py:    max_lines: 500
  backend/infra/**/*.py:           max_lines: 400
```

超限 → CI 挂。**这一条规则顶过 10 条架构文档**。

### 2.2 导入方向约束（防循环依赖 + 防破层）

使用 `import-linter`，在 `importlinter.ini` 中：

```ini
[importlinter:contract:layers]
name = moneybag 四层架构
type = layers
layers =
    api
    use_cases
    domain
    infra

[importlinter:contract:domain-services-no-crosstalk]
name = domain/services 之间禁止互相 import
type = forbidden
source_modules = domain.services
forbidden_modules = domain.services
```

效果：任何跨层反向依赖、任何 `domain/services/A` 直接 import `domain/services/B`，CI 红。

### 2.3 禁止的坏味道（grep 级防线）

CI 里加 5 行 shell 脚本，违规即挂：

| 禁止的模式 | 为什么禁 | 替代方式 |
|------------|----------|---------|
| `^_cache\s*=\s*\{\}`（模块级）| 必须走统一缓存 | `infra/cache` |
| `open(DATA_DIR` 裸读写 | 必须走持久化层 | `infra/store` |
| `import tushare` / `import akshare` 出现在 `domain/` 或 `api/` | 必须走数据源适配层 | `infra/data_source` |
| `httpx.post("https://api.deepseek` 直调 | 必须走 LLM 网关 | `infra/llm/gateway` |
| `MODULE_META\s*=\s*\{` | 已废弃的元数据底座 | 删除即可 |

这 5 条**覆盖 80% 的结构腐化场景**。CI 2 秒跑完。

### 2.4 每次 PR 自动输出架构体检报告

`scripts/architecture_report.py` 在 CI 中生成一段评论：

```
📊 架构体检（本次 PR）:
  main.py:               148 行 / 上限 200  ✅
  api/ 文件数:            11
  api/ 平均行数:          228 / 上限 400   ✅
  services 互相 import:   3 次（上次 3 次）✅
  模块级 _cache:          0 个              ✅
  裸读 DATA_DIR:          0 处              ✅
  未模板化的 prompt:      0 处              ✅
```

**原理**：把指标可见作为默认状态。每次 PR 都会看到，任一项恶化自然警觉。

## 三、主动设计：加新功能的 5 步标准流程

被动防御拦已知错误，主动流程预防未知错误。**每次加功能前，先过这 5 步**（写进 PR template）：

```
1. 这个功能属于哪一层？
   □ 只是展示数据           → 加 api/ 路由
   □ 编排多个服务            → 加 use_cases/
   □ 新业务规则 / 计算        → 加 domain/rule_engine 或 domain/services
   □ 接外部系统              → 加 infra/

2. 用到的现有能力走哪个接口？
   □ 缓存       → infra/cache
   □ 持久化     → infra/store
   □ 外部数据源  → infra/data_source
   □ LLM       → infra/llm/gateway
   □ 金融常识   → infra/knowledge（RAG）
   ⚠️ 想新写一个"类似但不完全一样"的东西？停。99% 是没找到现有接口。

3. 新引入的业务对象要不要进 domain/models？
   □ 有状态、多处使用           → 建 dataclass/Pydantic，进 domain/models
   □ 仅本地一次使用、不跨 >2 层  → 临时 dict 也可

4. 有没有新增硬阈值 / 魔法数字？
   □ 有 → 必须进 domain/rule_engine/defaults.py，禁止业务代码内联

5. 测试怎么写？
   □ 用到 infra 的地方必须可 Mock
   □ 至少一个能独立跑的单元测试（不依赖真实网络/文件）
```

**做成 PR template 的 checkbox，每个 PR 必填**。

## 四、"类似 / 已有"优先原则（最常见的腐化源）

加新功能前必须先问："**项目里有没有类似的已经做过？**"

- 有 → 照着写，一致性比"完美"更重要
- 没有 → 你是第一个做这类事的，谨慎设计

### 4.1 反腐化红线：禁止的命名

**这句话一出现就是腐化的开始**：
> "我知道有 `recommend_service`，但我的场景有点不一样，我新建一个 `recommend_service_v2` 吧。"

**绝不允许的命名后缀**：`_v2`、`_new`、`_ext`、`_helper`、`_utils2`、`_refactor`……

正确做法：
- **要么**重构现有 service，加参数 / 加策略让它支持新场景
- **要么**明确说"这是全新领域"，用**完全不同的名字**建新 service

（注：`02-code-audit.md` 提到的 `decision_maker_v2` 是重构过渡期的临时产物，M5 必须清理，之后再不允许 `_v2`。）

## 五、接口先行（先写 Protocol，再写实现）

新增 domain service 或 infra 组件的强制步骤：

1. 先在 `domain/protocols/xxx.py` 写一个 `Protocol`（20-30 行）
2. 再写具体实现（可能 200 行）

**好处**：
- 写 Protocol 的过程**逼你想清楚接口边界**
- 未来换实现、Mock 测试直接受益
- Protocol 本身就是**最好的接口文档**

## 六、新 LLM 调用必须走模板 + RAG

`04-ai-interface.md` 的字段级硬边界最容易被新功能破坏。规则：

- 所有 prompt 存在 `infra/llm/prompt_templates/*.md`，带版本号
- 所有 prompt 必须经过 `red_team_audit` 扫禁用词
- 所有"有观点"的输出必须引用 RAG 检索结果（`08-knowledge-rag.md`）
- 调用方式：`gateway.call(prompt_id="xxx", vars={...})`

**业务代码里硬编码 prompt 字符串 → CI 拒绝**（§2.3 有对应 grep 规则）。

## 七、"两次就抽象"原则

避免过早抽象（YAGNI）和永不抽象（复制粘贴）的折中：

| 写第几次 | 该做什么 |
|---------|---------|
| 第 1 次写某段逻辑 | 允许写在业务代码里，**留个 TODO 注释** |
| 第 2 次写一模一样的 | **必须抽象**到 infra 或 domain |
| 第 3 次及以后 | 如果还在复制粘贴 = 已经腐化，CR 打回 |

**原理**：第一次抽象往往抽错（需求还不够清晰），第三次抽象已经迟了。**第二次是黄金时机**。

## 九、新功能合并前的三件事（PR 必填）

每个功能 PR merge 前三件事**缺一不可**：

1. **单元测试** —— 至少 1 个，依赖 infra 的必须能 Mock
2. **更新 `docs/ARCHITECTURE.md`** —— 如果新增了模块/接口/Protocol
3. **更新 `CHANGELOG.md`** —— 一句话说明做了什么

PR template 把这三条列成 checkbox，没打勾不让 merge。

## 十一、最易腐化的三处 —— 重点防守

基于当前代码状态，未来最可能再次烂掉的地方：

### 🔴 1. main.py 拆完后会慢慢变胖

**典型腐化场景**：加个小接口嫌拆 router 麻烦，塞 main.py 一下下 → 三个月后又 1000 行。

**防守**：§2.1 CI 强制 main.py <200 行，超了挂。没有侥幸空间。

### 🔴 2. `_cache = {}` 又悄悄出现

**典型腐化场景**：写新功能时想加个快速缓存，3 行代码解决，懒得走 `infra/cache`。

**防守**：§2.3 CI grep 检测，违规挂。**3 行的方便永远不值**。

### 🔴 3. LLM prompt 又硬编码到业务代码

**典型腐化场景**：新功能想试个新 prompt，业务代码里拼字符串，没走模板。

**防守**：§2.3 CI grep `messages=\[` / `"role":\s*"system"` 在 `domain/` 和 `api/` 下出现即挂。

## 十二甲、影响面清单（Impact Report）—— 对抗 AI/人的上下文盲区

> 📌 **配合 `00-ANCHOR.md` "🛠 改动三步法" 使用**：三步法讲**流程**（改前查表 → 改中搜索 → 改后出清单），本节讲**第 3 步的交付格式**。两者合起来是完整闭环。

**核心问题**：无论是 AI 还是人类，单独处理一个模块时容易忘记上下游影响。"改完就合"是耦合崩坏的主要入口。

**解决**：**每次改动强制产出影响清单**。AI 协作时，结束前必须主动生成；人类开发时，PR description 必填。

### 12甲.1 AI 协作结束的交付格式

每次 AI 完成任务时（无论改代码还是改文档），**结束前主动产出**以下格式：

```markdown
## ✅ 本次改动

**改了什么**：<一句话>
**改在哪**：<文件名 + 行号/章节>

## 📢 影响面清单

**对照 `00-ANCHOR.md` 改动传播表**，本次改动可能波及：

### 🔴 必须同步评估（应改未改 = 耦合崩）
- <文件 A>：因为 <原因>，建议检查 <具体位置>
- <文件 B>：...

### 🟡 建议评估（可能无影响，但值得看一眼）
- <文件 C>：...

### 🟢 确认无影响（列出来避免漏查）
- <文件 D>、<文件 E>

## ❓ 需要你决定的事

- <悬而未决的问题 1>
- <悬而未决的问题 2>

## 🗺️ 建议的下一步

- 下次会话建议贴：<文件清单>
- 暂时不急的：<文件清单>
```

### 12甲.2 人类 PR 的影响清单

PR description 里填上面"影响面清单"小节内容。**CI 检查 PR 正文必须包含"## 📢 影响面清单" 字样**，否则挡下。

### 12甲.3 影响清单的三级分类

| 级别 | 含义 | 必须处理吗 |
|-----|------|-----------|
| 🔴 必须同步评估 | 改了接口/字段/阈值，下游一定受影响 | **是**。本 PR 不处理就挂 TODO，说明原因 |
| 🟡 建议评估 | 可能受影响，要看一眼 | 否。写"已检查无影响"即可 |
| 🟢 确认无影响 | 已查过、无影响 | 否。写出来是为了**证明你查了**，不是漏查 |

### 12甲.4 为什么这招有效

1. **AI 结束时主动产出** → 不再需要你"审完才发现漏了 X"
2. **分三级** → 防止影响清单长成"流水账"
3. **强制列 🟢** → 看似啰嗦，实则是防漏查的最大保险。"我查过了没事"和"我忘了查"完全是两回事
4. **和 ANCHOR 传播表联动** → 不用凭空想，查表对号入座

### 12甲.5 反例：没有影响清单会怎样

**场景 1：改 03 阈值，漏同步 07**
- 用户看到 07 清单第 7 条文案 "偏离 >10% 建议再平衡"，但系统阈值已改为 7%。系统提示了用户没看到的数字。
- **影响清单会拦住**：改 03 时必列 07 为 🔴。

**场景 2：改 04 LLM 返回结构，漏同步 05 缓存 key**
- 05 缓存 key 没变但内容变了，前端读旧格式崩。
- **影响清单会拦住**：改 04 返回格式时必列 05 为 🔴。

**场景 3：删 08 某篇 RAG 文章，漏看 09 周度小课**
- 09 小课内容引用了那篇文章，推送时引用 404。
- **影响清单会拦住**：改 08 内容时必列 09 为 🟡。
