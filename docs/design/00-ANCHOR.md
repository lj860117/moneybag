## 📍 给协作 AI / 开发者的使用说明（最重要）

### 每次任务开始前必做

1. **读本文件**（00-ANCHOR.md）—— 锁定不变式、看任务对照表
2. **按对照表贴 1-3 个相关文件** —— 不要凭记忆做事
3. **不确定的内容 → 问，不要猜** —— 没在文件里的东西不存在
4. **跨文件改动要明说** —— 改 06 影响 03 的，请一起贴

### 每次任务结束前必做 ⭐

**产出"影响面清单"**（格式见 `13-governance.md` §十二甲）。AI 结束回复前必须主动产出以下结构：

```
✅ 本次改动：<改了什么 + 改在哪>

📢 影响面清单（对照 ANCHOR 改动传播表）：
🔴 必须同步评估：<文件列表 + 原因>（应改未改 = 耦合崩）
🟡 建议评估：<文件列表>（可能无影响，看一眼）
🟢 确认无影响：<文件列表>（证明你查了）

❓ 需要你决定：<悬而未决的问题>
🗺️ 下一步建议：<下次会话贴哪些文件>
```

**没列影响面清单的回复 = 未完成**。用户可以直接追问："影响面清单呢？"

### 绝对禁止

- ❌ 凭印象说"按 §X.Y 所说"—— 必须贴具体文件
- ❌ 自由发挥"类似的做法是……"—— 只执行文件里写了的
- ❌ 一次读 5+ 文件 —— 分次会话做
- ❌ **结束前不产出影响面清单** —— 这是硬性要求

### 上下文不足时

如果任务涉及 4+ 个文件：**拆成多次会话**，每次 2-3 个文件。别硬塞。

## 🔒 12 条不变式（Invariants，永不修改）

**每条都是 CI/CD 会拦截的硬规则。破了这些 = 破了整个设计。**

1. **AI 不预测证券价格** —— 不输出 action / position_pct / 未来收益率
2. **AI 可对"框架"做判断，不能对"证券"做预测** —— 股债比可以，选股不行
3. **所有 LLM 调用走 `infra/llm/gateway`** —— 不允许业务代码直调 httpx
4. **所有缓存走 `infra/cache`** —— 禁止模块级 `_cache = {}`
5. **所有文件 IO 走 `infra/store`** —— 禁止裸读 `DATA_DIR`
6. **所有外部数据源走 `infra/data_source`** —— 不允许 domain/api 层 import tushare
7. **禁止 `_v2` / `_new` / `_ext` / `_helper` 命名** —— 要么重构要么新领域新名
8. **`main.py` < 200 行** —— 超了 CI 挂
9. **`domain/services` 之间禁止互相 import** —— 只能走 `domain/protocols`
10. **单向依赖**：`api/` → `use_cases/` → `domain/` → `infra/`，反向禁止
11. **M1 之后，新建跨模块接口必须先写 Protocol**（方案 C）—— 文档契约只是 M1 前的过渡草稿
12. **每次改动必须产出"影响面清单"** —— AI 结束前必出，人类 PR 必填（见 `13-governance.md` §十二甲）

## ⚠️ 方案 C 激活提醒（每次开会请检查）

**背景**：当前是"方案 A+B 过渡期"（纯文档约束），M1 W1 起切到"方案 C 终局"（Protocol + CI 代码约束）。

**4 阶段执行**：

| 阶段 | 触发时机 | 动作 |
|------|---------|------|
| 1 | 现在～M1 启动 | **方案 A+B 过渡**：每个文件顶部有"模块契约"头部 + 本文件"改动传播表" |
| 2 | **M1 W1 开工时** 🔔 | **启动方案 C**：写 4 个核心 Protocol + 配 import-linter + mypy + CI |
| 3 | M1-M2 推进中 | 文档契约头部**逐步瘦身**为"见 XxxProtocol"，细节下沉到代码 |
| 4 | 每季度体检 | 检查新增模块是否都走 Protocol，不再只写文档契约 |

**触发关键词**（AI 看到任一必须主动提醒方案 C）：
- "M1 W1" / "开始搭骨架" / "搭四层骨架"
- "infra/cache" / "infra/store" / "infra/llm/gateway" 首次实现
- "配 CI" / "import-linter" / "mypy"
- "新建接口" / "新 Protocol" / "跨模块调用"

**如果你看见 AI 该提醒却没提醒，请直接追问**："是不是该启动方案 C 了？"

## 🚨 改动传播表（改 A 必须同步评估 B）

> **用途**：你准备**改动**某个文件/模块前，查本表确认"会波及哪些下游"。
> **和上方"阅读依赖图"的区别**：阅读依赖讲"先读什么"；改动传播讲"改了影响谁"。方向相反，都要看。
> **配合 §不变式 12**：每次改动结束必须产出"影响面清单"，本表是写清单时的查询工具。

**用途**：动手改任何一个文件/模块前，先查本表，看有没有下游要同步改。漏同步是模块耦合最大的崩坏源。

| 改动位置 | 必须同步评估 | 为什么 |
|---------|-------------|-------|
| **03 规则引擎阈值**（配比矩阵/偏离度/风险规则）| 07（7 点清单引用阈值）| 清单每条规则的判断来自 03 的阈值 |
| **03 规则引擎接口**（函数签名/返回字段）| 07 + 04 + 05 | 清单/AI 解读/凌晨工厂都调用 |
| **03 defaults.py 常量** | 检查是否业务代码直接用了常量名 | 常量改名会 import 失败 |
| **06 FamilyProfile 字段**（新增/删除/改名）| 03（规则引擎输入）+ 07（清单第 1/2 条依赖画像）| 画像字段是规则输入 |
| **06 资产负债表字段**（Tier 1/2）| 03 应急金规则 + 07 清单 + 05 实时事实层 | 多方消费 |

| **04 LLM 返回 JSON 结构** | 05 缓存 key + 08 RAG 引用字段 + 前端 UI | 下游全吃 JSON |
| **04 禁用词正则** | 13 CI 规则（同步 grep 脚本）| CI 和 runtime 必须一致 |
| **04 对话锚点规则** | 前端入口收敛、05 缓存 | 无锚点 chat M3 下线 |

| **05 event 触发器** | 06 交易录入 / 7 点清单触发 | A 层实时重算 |
| **07 7 点清单新增/删除一条** | 03 规则引擎（阈值）+ 06 资产负债表（字段）+ 决策日志 schema | 变更规则数量影响档案 |
| **07 买入理由选项** | 决策日志 schema + M5 复盘归因逻辑 | 归因统计依赖选项枚举 |
| **08 RAG 文档格式** | 04 prompt 模板引用字段 + 09 苏格拉底提问 | 下游检索后字段固定 |
| **08 文章增删** | 07 清单的"延伸阅读" + 09 周度小课推送 | 内容源统一管理 |

| **12 四层目录结构** | 所有模块（api 路径/domain import）| 底座动则天动 |
| **12 Protocol 签名** | 所有实现类（domain/services + infra 实现）| M1 W1 后，Protocol 是硬契约 |
| **12 infra/cache 接口** | 迁移期所有用到缓存的地方 | 必须并存双写 1 周再切 |
| **13 CI 规则** | 本文件不变式 + 每个文件的"禁止的坏味道" | 代码层和文档层必须一致 |
| **00 不变式新增** | 考虑是否需要加 CI 规则落地 | 不变式不是口号，必须有 CI 强制 |

### 使用方式

改任何一处之前：
1. 在本表查"改动位置"列
2. 看"必须同步评估"列列出了哪些文件
3. 评估这些文件的影响范围（可能只是确认没影响，也可能真要改）
4. 写 PR description 时列出"本次影响到的下游"

### 漏了会怎样

- 改 03 阈值 → 忘了 07：**清单判断标准对不上**，用户看到"应急金不足"的定义和系统阈值不一致
- 改 06 字段 → 忘了 03：**规则引擎报 KeyError**
- 改 04 返回结构 → 忘了 05：**缓存 key 不变但内容变了**，前端渲染崩
- 改 08 RAG 格式 → 忘了 04：**AI prompt 填错字段**，生成垃圾文案

**所有"看起来独立"的改动，99% 有下游要同步**。

## 🛠 改动三步法（配合不变式 12）

> **宏观 → 微观 → 结果**，三步全做完才算改完。漏任一步 = 改动有隐患。

### 第 1 步：改前（查传播表）

- 打开上方 **"🚨 改动传播表"**
- 找到你要改的位置
- 记下 **"必须同步评估"** 列的所有文件
- 任务指令里**把这些文件全部贴上**，不能只贴改动的那个

### 第 2 步：改中（全局搜索 + 代码扫描）

改动前必须做的**微观实操**（跳过这一步 = 改完必漏）：

- [ ] 全局搜索旧**字段名 / 常量名 / 接口签名** → 列出所有引用点
- [ ] 确认每个引用点**是否需要同步修改**（不是全部都要改，但每个都必须**评估过**）
- [ ] 改了 **JSON 结构 → `infra/cache` key 必须变**（key 不变 + 内容变 = 缓存脏数据）
- [ ] 改了**规则引擎返回字段** → `05-scheduling.md` 三层缓存 key 同步评估
- [ ] 改了**前端消费字段** → 对应 `api/*.py` 的 response schema 要同步

### 第 3 步：改后（产出影响清单）

见 **不变式 12**。AI 结束回复前必出"✅ 本次改动 / 📢 影响面清单 / ❓ 需要决定 / 🗺️ 下一步建议"。

## 🚫 改动禁止事项

- ❌ 只改声明处不改引用处（导致运行时 KeyError / 接口报错）
- ❌ 改 JSON 结构不改缓存 key（导致缓存脏数据）
- ❌ 跳过第 2 步直接改（导致漏改引用）
- ❌ 改完不产出影响清单（违反不变式 12）
- ❌ 一次改 5+ 文件（拆成多次会话，每次 2-3 文件）
