INVESTIGATION: Why rule-based replies never trigger, always using LLM

================================================================================
KEY FINDINGS
================================================================================

🔴 CRITICAL ISSUE: RULES EXIST BUT NEVER TRIGGERED

The rule-based reply system IS implemented and functional, but due to an 
architectural design flaw, it's NEVER called when the LLM API key works.

Current behavior:
  ❌ Always tries LLM first (7-15 seconds latency)
  ❌ Only falls back to rules if LLM fails (rare)
  ❌ Rules never called for normal successful requests

Expected behavior (per code comments):
  ✅ Should use rules first for common patterns (< 1 second)
  ✅ Should only call LLM for complex questions (7-15 seconds)

Impact:
  • Performance: 10-15x slower than intended
  • User experience: Unnecessary waiting on simple questions
  • Cost: Unnecessary LLM API calls

================================================================================
ROOT CAUSE
================================================================================

SMOKING GUN: Line 45 in chat.py says "规则优先，不调 LLM" (Rules FIRST, NO LLM)
But line 80 immediately calls LLM anyway.

This indicates INCOMPLETE REFACTORING.

The developer intended to:
  1. Classify intent
  2. If common pattern → use rules (fast)
  3. If complex → use LLM

But only implemented:
  1. Classify intent
  2. Always use LLM (fast path never implemented)
  3. Fallback to rules only if LLM fails

DETAILED FINDINGS
================================================================================

Q1: Is _rule_based_reply() actually called?
A1: ONLY if:
    • LLM API key is missing, OR
    • LLM request returns HTTP error (status != 200)
    
    NEVER called when LLM works successfully.

Q2: What triggers rule-based responses?
A2: Fallback only (lines 341, 365 in chat.py):
    • Streaming endpoint: if LLM request fails
    • Non-streaming endpoint: if LLM request fails
    
    NO primary path for rules.

Q3: What patterns can rules handle?
A3: 15+ patterns with real-time market data:
    ✓ Timing/entry (入场时机)
    ✓ Take profit/stop loss (止盈止损)
    ✓ Smart DCA (定投)
    ✓ Market sentiment (跌/涨)
    ✓ Specific assets (黄金, 标普, etc.)
    ✓ Macro/policy/news (央行, 政策)
    ✓ Any other question (fallback)
    
    All use real-time data from data_layer.py

Q4: Does the rule engine work?
A4: YES, fully implemented (400+ lines in shared_helpers.py):
    • Keyword matching: 15+ specific patterns
    • Real-time data: ✓ Fear & Greed Index
    •                 ✓ Valuation percentile
    •                 ✓ Technical indicators
    •                 ✓ Fund prices & news
    •                 ✓ Policy & macro data
    • Response generation: Deterministic, fast (<100ms)

Q5: Do both endpoints have the same problem?
A5: YES, both always prioritize LLM:
    • /api/chat (non-streaming): LLM first, rules fallback
    • /api/chat/stream (streaming): LLM first, rules fallback
    
    BOTH have incomplete rules-first implementation.
