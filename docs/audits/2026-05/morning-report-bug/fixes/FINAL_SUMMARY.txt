================================================================================
PROBLEM STATEMENT
================================================================================

Users were seeing morning reports with incorrect dates:
- Reported Date: "2025年7月14日" 
- Actual Current Date: "2026年5月14日"
- Duration of Issue: Unknown, but systematic (affects any date mismatch)

Root Cause:
The briefing() method in steward.py only checked if a cache file EXISTS,
without validating that the file's creation date matched the current date.
When old cache files remained in the system, they would be served indefinitely.

Impact:
- Users received stale dated reports
- Historical briefing lists contained incorrect dates
- Cache pollution degraded system reliability

================================================================================
SOLUTION IMPLEMENTED
================================================================================

Five core changes to backend/services/steward.py:

1. IMPORT: Added timedelta for date arithmetic
   Line 14: from datetime import datetime, timedelta

2. HELPER: _extract_cache_date(filename: str) -> str
   Lines 49-58
   Purpose: Parse YYYYMMDD date from filename "{user_id}_{YYYYMMDD}.json"
   
3. HELPER: _check_date_consistency() -> bool
   Lines 38-46
   Purpose: Validate system date is in range [2020, 2050]
   
4. MODIFIED: briefing() method
   Lines 153-214
   Changes:
   - Extracts date from cache filename
   - Compares with current date
   - Deletes cache if date doesn't match
   - Generates fresh report if needed
   
5. MODIFIED: briefing_history() method
   Lines 288-331
   Changes:
   - Validates date format (8 digit YYYYMMDD)
   - Filters out future-dated files
   - Filters out files older than N days
   - Returns only valid recent histories
