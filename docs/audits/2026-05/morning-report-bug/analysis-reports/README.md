## 📊 问题概览

### 严重性分布
| 等级 | 问题数 | 用户感受 | 修复时间 |
|------|--------|---------|---------|
| 🚨 P1（致命） | 2 | 完全收不到晨报 | 1 小时 |
| ⚠️ P2（严重） | 2 | 收到过期的坏数据 | 1 小时 |
| ⚠️ P3（重要） | 2 | 分析精度下降 | 1.5 小时 |
| ℹ️ P4（优化） | 4 | 缺少告警和监控 | 2.5 小时 |

### 根本原因（3 个）
1. **错误吞噬模式**：数据源失败返回 None 而非异常，异常被逐层吞噬
2. **缓存穿透漏洞**：坏数据被缓存锁定一整天，即使数据源已恢复
3. **推送中止点无保护**：磁盘满、权限错误时脚本直接退出，无告警

## 📞 常见问题

**Q: 哪些修复是必须的？**
A: P1 和 P2 修复是必须的（2 小时）。否则用户要么收不到晨报，要么收到过期的坏数据。

**Q: 可以只修复 P1 不修 P2 吗？**
A: 不建议。P1 修复保障推送不失败，但用户仍然可能收到"暂无分析"。P2 修复确保数据质量。

**Q: 修复需要修改数据库吗？**
A: 不需要。所有修复都是代码级别的，只需要修改代码文件。

**Q: 修复会破坏现有功能吗？**
A: 不会。所有修复都是增加保护和告警，没有改变核心逻辑。

**Q: 需要重启服务吗？**
A: 是的。修改后需要重启后端服务和 night_worker 脚本。
