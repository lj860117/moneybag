## 一、启动清单（每次新 AI 会话必执行）

> AI 收到第一条实质任务时，按顺序执行，全部完成前禁止写代码/改文件/跑 git。
> 违反 = P0 事故。

### 第零步：记忆健康检查（3秒）

1. 日志文件 > 30 天 → 自动蒸馏到 MEMORY.md，删除原文件
2. MEMORY.md > 100 行 → 提醒拆分
3. 检查矛盾记忆（旧路径 vs 新路径、已删 Skill 仍在列表）
4. 发现问题 → 静默修复简单矛盾，复杂问题一行提醒

### 第一步：读取记忆（5秒）

1. 读 `SESSION-STATE.md` — **最优先**，恢复上次会话的任务状态
2. 读 `MEMORY.md` — 了解环境、约定、待办
3. 读今天的日志 — 了解今天已做了什么
4. 读昨天的日志 — 了解上下文连续性
5. 按需读偏好/同步规则
6. 检查未同步改动 → 提醒补推

### 第二步：识别任务 → 路由 Skill（10秒）

1. 根据用户请求匹配对应 Skill
2. 匹配到 → **实际调用加载**，不是只"知道有"
3. 匹配不到 → 询问用户
4. moneybag 项目专属：必须先读踩坑教训 #18（后端做了前端必须接）+ #19（多ID统一入口）
5. Skill 差异检测：本地未安装 → 提醒用户并给安装命令

### 第三步：Skill 过期检查（3秒）

- 巡检 > 10 天 或 调研 > 35 天 → 一行提醒
- 紧急 bug 修复时跳过，完成后再提
- 每周巡检时同时执行保密审计

### 第四步：开始干活

- 遵守铁律（见第二章）
- 遵守对应 Skill 的规范
- 改完代码提醒验证
- **WAL 协议**（预写日志）→ 检测到关键信号时先写文件再回复
- 做完实质工作 → 写当日日志

### 第五步：收尾闭环

- 检查是否有内容需要提炼到长期记忆
- 有未完成任务 → 写入 MEMORY.md 待恢复区
- 更新 SESSION-STATE.md → 清空当前任务、记录摘要

## 二、铁律（20条，绝对不能违反）

### 🔴 核心铁律

| # | 铁律 | 要点 |
|:---:|---|---|
| 1 | **绝不用正则做批量重构** | 手工逐文件替换 |
| 2 | **改代码前先确认可回滚** | 告知即将修改的文件清单 |
| 3 | **改完一个文件立即验证** | 不攒改动 |
| 4 | **超过2轮修不好就停** | 告诉用户需要回滚，不越修越烂 |
| 5 | **linter ≠ 编译器** | 不以 VSCode 零错误作为"修好了"的依据 |
| 6 | **涉及技术深度先查参考** | 不凭空写算法 |
| 7 | **方案要有出处** | 关键决策说得出"谁用过" |
| 8 | **最小可用版本先交** | 核心功能先跑通再迭代 |
| 9 | **超出能力范围就说** | 建议用现成库 |
| 10 | **记忆 ≠ 事实** | MEMORY 可能过时，涉及路径/配置/API 变更时先验证 |
| 11 | **删除文件必须三确认** | ① 文件清单 ② 风险说明 ③ 用户确认。走回收站，每批≤10个 |
| 12 | **改完必须只读验证** | 重读所有改动文件确认写入正确，检查关联文件遗漏 |
| 13 | **"不能"=穷尽所有选项** | 至少尝试3-5种替代方案再说"做不到" |
| 14 | **SKILL.md 只放规则，知识靠引用** | 操作手册/模板/示例放 references/ |
| 15 | **SKILL.md 写入前预算检查** | wc -l: ≤1300 OK / >1500 禁止 |
| 16 | **Git 推送分两档** | 收工=完整同步 / 随手=只推Skill |
| 17 | **启动清单是硬门禁** | 收到第一条消息必须先跑清单，完成前禁止一切操作。**P0事故教训** |
| 18 | **后端API做了必须验前端调了** | grep 前端确认有调用。来源：政策5主题做了后端前端0调用 |
| 19 | **多ID系统统一入口函数** | 不允许两个函数返回不同ID。来源：getUserId vs getProfileId 导致18处隔离破裂 |
| 20 | **推送先查格式兼容性** | 不假设支持Markdown。来源：企微Markdown在微信端显示"不支持此消息类型" |

### 🔴 P0 事故档案

**#001：V5.0 启动清单跳过事故（2026-04-12）**

- **影响**：V5.0 全部4个新功能 + 14模块拆分，共2000+行代码未经规范检查 push 到生产
- **根因**：会话恢复时 AI 直接继续写代码，完全跳过启动清单5个步骤
- **产生的铁律**：#17（启动清单硬门禁）
- **防御机制**：铁律 #17 + SOUL.md 硬指令双保险

## 三、WAL 协议（预写日志 — 先写后答）

> 借鉴数据库事务思路：检测到关键信号 → 先落盘到文件 → 再回复用户。

### 6种关键信号

| 信号类型 | 识别模式 | 写入位置 |
|---------|---------|---------|
| 🔄 **用户纠正** | "不对"/"其实"/"错了" | preferences.md 或 日志踩坑 |
| 📛 **专有名词** | 首次出现的项目名/人名/工具名 | 日志关键决策 或 MEMORY |
| 💭 **偏好表达** | "我喜欢"/"我习惯"/"以后都这样" | preferences.md（立即更新） |
| ⚖️ **重要决策** | "用X吧"/"选Y"/"就这样定了" | 日志关键决策 |
| 📝 **内容变更** | "把XX改成YY"/"那段删了" | 日志改动记录 |
| 🔢 **具体数值** | 首次提到的日期/金额/ID/URL/版本号 | 日志 或 MEMORY |

**执行规则**：信号匹配 → 写入文件 → 继续回复（用户无感知）。不确定 → 宁可多记。

## 六、Memory vs Skill 分类标准

> Memory 存"事实"，Skill 存"方法"。

| 存什么 | 存哪里 | 判断标准 | 示例 |
|--------|--------|---------|------|
| 事实 | MEMORY / 日志 / preferences | 背景信息、一次性决策、偏好、状态 | 项目路径、Git身份、技术选型 |
| 方法 | SKILL / references | 可复用流程、标准操作步骤 | 闭环流程、检查清单 |
| 踩坑 | 日志 → 提炼为铁律 | 可泛化为"以后都要这样做" | GBK编码坑→铁律 |
| 灵感 | 日志（仅Memory） | 还未验证的想法 | "可以做XX功能" |

**三问速判**：
1. "知道就行" vs "要按这个做"？ → 前者Memory，后者Skill
2. 换个会话还需遵守？ → 需要→Skill
3. 只跟当前状态有关？ → Memory

## 八、SKILL.md 瘦身铁律

> 研究依据：Stanford Lost-in-the-Middle + Anthropic 建议 + 社区共识

1. 主文件上限 500 行（>400 警惕，>500 禁止）
2. 新增内容默认写 references/（只有"每次必须遵守的规则"才进主文件）
3. 30 秒可扫描原则
4. 示例代码/操作手册/踩坑表 不进主文件
5. 每次写入前 wc -l 检查
6. **此规则适用于所有 Skill 和项目文档** — 通用知识架构原则

以下内容**不得**出现在 Git 仓库中：
- 公司项目细节（项目名/代号、功能描述、业务逻辑）
- P4 服务器地址、公司内部路径
- 团队成员信息（同事名字、工号）
- 公司引擎/内部 API 名称

**判断标准**：不确定是否敏感？→ **删掉**。

## 十二、moneybag 踩坑教训（实战总结，跨会话复用）

| # | 教训 | 防范 |
|---|------|------|
| 1 | 东方财富反爬是常态 | 必须多源降级（新浪+雪球） |
| 2 | NaN/Inf 必须递归清洗 | json.dumps 前过 `_clean_nan()` |
| 3 | 前后端 key 必须对齐 | 后端中文key前端英文key是必坑 |
| 4 | getUserId 和 getProfileId 必须统一 | 两函数返回不同ID导致数据存错 |
| 5 | 企微 Markdown 在微信端不可用 | 只能用纯文本推送 |
| 6 | 企微 IP 白名单需先配回调 URL | 回调接口必须先部署好 |
| 7 | replace_in_file 容易误删代码 | 改大文件时 old_str 包含足够上下文 |
| 8 | main.py 膨胀必须及时拆 | 超2000行就该拆 router |
| 9 | 后端做了前端没接 = 白做 | 每次做后端API后检查前端是否调用 |
| 10 | 模板字符串嵌套 IIFE 会炸 | app.js 里用辅助函数替代 |
